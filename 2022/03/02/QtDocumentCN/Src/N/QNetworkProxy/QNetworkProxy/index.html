详细描述#

QNetworkProxy提供了为Qt网络类配置网络层代理支持的方法。目前支持的类有 QAbstractSocket、QTcpSocket、QUdpSocket、QTcpServer 和 QNetworkAccessManager 。代理支持被设计为尽可能的透明。这意味着你编写的现有的支持网络的应用程序应该使用以下代码自动支持网络代理。

QNetworkProxy proxy;
proxy.setType(QNetworkProxy::Socks5Proxy);
proxy.setHostName("proxy.example.com");
proxy.setPort(1080);
proxy.setUser("username");
proxy.setPassword("password");
QNetworkProxy::setApplicationProxy(proxy);

设置应用程序范围内的代理的另一种方法是使用QAbstractSocket::setProxy()和QTcpServer::setProxy()为单个套接字指定代理。这样一来，就可以用下面的代码来禁止对特定的套接字使用代理。

serverSocket->setProxy(QNetworkProxy::NoProxy);

如果connectToHost()、bind()或listen()中使用的地址等同于QHostAddress::LocalHost或QHostAddress::LocalHostIPv6，则不会使用网络代理。

每种类型的代理支持都有一些相关的限制。在选择使用的代理类型之前，你应该仔细阅读ProxyType文档。

注意：对当前连接的套接字进行的更改不会生效。如果你需要改变一个连接的套接字，你应该重新连接它。

SOCKS5#

从Qt 4开始的SOCKS5支持是基于RFC 1928和RFC 1929。支持的认证方法是无认证和用户名/密码认证。IPv4和IPv6都被支持。如果QNetworkProxy::HostNameLookupCapability被启用，域名将通过SOCKS5服务器解析，否则将在本地解析，并将IP地址发送到服务器上。当用QUdpSocket和QTcpServer使用SOCKS5时，有几件事需要记住。

使用QUdpSocket时，对bind()的调用可能会因超时错误而失败。如果向 bind()传递了 0 以外的端口号，不保证会使用指定的端口。使用 localPort() 和 localAddress() 来获取实际使用的地址和端口号。因为代理的UDP要经过两个UDP连接，所以数据包更有可能被丢弃。

使用QTcpServer调用listen()可能会因超时错误而失败。如果向listen()传递了一个0以外的端口号，那么不保证会使用指定的端口。使用serverPort()和serverAddress()来获取用于监听连接的实际地址和端口。SOCKS5每次调用listen()只支持一个被接受的连接，而且每次调用可能会导致使用不同的serverPort()。

See also QAbstractSocket and QTcpServer.

成员类型文档#

enum QNetworkProxy::Capability flags QNetworkProxy::Capabilities#

这些标志表示一个特定的代理服务器所支持的能力。

QNetworkProxy在创建对象时默认设置了不同的能力（参见QNetworkProxy::ProxyType中的默认值列表）。然而，在对象被创建后，可以通过setCapabilities()来改变能力。

QNetworkProxy]()支持的能力有:

Constant Value Description
QNetworkProxy::TunnelingCapability 0x0001 能够打开到远程主机的透明、隧道式的TCP连接。代理服务器逐字逐句地从一方转发到另一方，不做任何缓存。
QNetworkProxy::ListeningCapability 0x0002 能够创建一个监听套接字，并等待从远程主机传入的TCP连接。
QNetworkProxy::UdpTunnelingCapability 0x0004 能够通过代理服务器转发UDP数据报，并从远程主机转发。
QNetworkProxy::CachingCapability 0x0008 缓存传输内容的能力。这种能力是针对每个协议和代理类型的。例如，HTTP代理可以缓存用 “GET “命令传输的网络数据的内容。
QNetworkProxy::HostNameLookupCapability 0x0010 能够连接执行对远程主机名称的查询并连接到它，而不是要求应用程序执行名称查询并仅请求连接到IP地址
QNetworkProxy::SctpTunnelingCapability 0x00020 能够打开与远程主机的透明、隧道式SCTP连接。
QNetworkProxy::SctpListeningCapability 0x00040 能够创建一个监听套接字，并等待从远程主机传入的SCTP连接。

Capabilities类型是QFlags的一个类型定义。它存储了Capability值的OR组合。

enum QNetworkProxy::ProxyType#

这个枚举描述了Qt中提供的网络代理的类型。

有两种类型的代理被Qt所理解：透明代理和缓存代理。第一类包括可以处理任何任意数据传输的代理，而第二类只能处理特定的请求。缓存代理只对可以使用它们的特定类有意义。

Constant Value Description
QNetworkProxy::NoProxy 2 不使用代理服务
QNetworkProxy::DefaultProxy 0 代理是根据使用setApplicationProxy()设置的应用程序代理来确定的。
QNetworkProxy::Socks5Proxy 1 Socks5使用代理服务
QNetworkProxy::HttpProxy 3 使用了HTTP透明代理
QNetworkProxy::HttpCachingProxy 4 仅为HTTP请求提供代理服务
QNetworkProxy::FtpCachingProxy 5 仅为FTP请求提供代理服务

下表列出了不同的代理类型和它们的能力。由于每种代理类型都有不同的能力，在选择代理类型之前，了解它们是很重要的。

Proxy type Description Default capabilities
SOCKS 5 适用于任何类型的连接的通用代理。支持TCP、UDP、与端口绑定（传入连接）和认证。 TunnelingCapability, ListeningCapability, UdpTunnelingCapability, HostNameLookupCapability
HTTP 使用 “CONNECT “命令实现，只支持传出的TCP连接；支持认证。 TunnelingCapability, CachingCapability, HostNameLookupCapability
Caching-only HTTP 使用普通的HTTP命令实现，它只在HTTP请求的背景下有用（见QNetworkAccessManager CachingCapability, HostNameLookupCapability
Caching FTP 使用一个FTP代理实现，它只在FTP请求的情况下有用（见QNetworkAccessManager CachingCapability, HostNameLookupCapability

还要注意的是，你不应该将应用程序的默认代理（setApplicationProxy()）设置为一个不具备TunnelingCapability能力的代理。如果你这样做，QTcpSocket将不知道如何打开连接。

See also setType(), type(), capabilities(), and setCapabilities().

bool QNetworkProxy::isCachingProxy() const#

如果这个代理支持QNetworkProxy::CachingCapability能力，则返回true。

在Qt 4.4中，该能力与代理类型绑定，但自Qt 4.5以来，可以通过调用setCapabilities()来移除代理的缓存能力。

See also capabilities(), type(), and isTransparentProxy().

bool QNetworkProxy::isTransparentProxy() const#

如果该代理支持TCP连接的透明隧道，则返回true。这与QNetworkProxy::TunnelingCapability的能力相符。

在Qt 4.4中，该能力与代理类型绑定，但自Qt 4.5以来，可以通过调用setCapabilities()来移除代理的缓存能力。

See also capabilities(), type(), and isCachingProxy().

[since 5.0] QNetworkProxy::rawHeader(const &headerName) const#

返回头文件headerName的原始形式。如果没有这样的头，或者代理不是HttpProxy或HttpCachingProxy类型，则返回一个空的QByteArray，这可能与一个存在但没有内容的头无法区分（使用hasRawHeader() 来找出头是否存在）。

原始头可以用setRawHeader()或setHeader()来设置。

这个函数是在Qt 5.0中引入的。

See also header() and setRawHeader().

[since 5.0]<> QNetworkProxy::rawHeaderList() const#

返回该网络代理中设置的所有原始头信息的列表。该列表是按照头信息被设置的顺序排列的。

如果该代理不是HttpProxy或HttpCachingProxy类型，将返回一个空的QList。

这个函数是在Qt 5.0中引入的。

See also hasRawHeader() and rawHeader().

[static]void QNetworkProxy::setApplicationProxy(const &networkProxy)#

将应用程序级别的网络代理设置为networkProxy。

如果QAbstractSocket或QTcpSocket有QNetworkProxy::DefaultProxy类型，那么就会使用用这个函数设置的QNetworkProxy。如果你想更灵活地决定使用哪个代理，请使用QNetworkProxyFactory 类。

用这个函数设置一个默认的代理值将覆盖用QNetworkProxyFactory::setApplicationProxyFactory设置的应用代理工厂，并禁止使用系统代理。

See also QNetworkProxyFactory, applicationProxy(), QAbstractSocket::setProxy(), and QTcpServer::setProxy().

[since 5.0]void QNetworkProxy::setRawHeader(const &headerName, const &headerValue)#

设置头信息headerName的值为headerValue。如果headerName对应于一个已知的头（见 QNetworkRequest::KnownHeaders），原始格式将被解析，相应的 “cooked”头也将被设置。

比如说:

request.setRawHeader(QByteArray("Last-Modified"), QByteArray("Sun, 06 Nov 1994 08:49:37 GMT"));

也会把已知的头LastModifiedHeader设置为解析后的日期的QDateTime对象。

注意：两次设置相同的头信息会覆盖之前的设置。为了完成同名的多个HTTP头的行为，你应该把两个值连接起来，用逗号（”,”）分开，然后设置一个单一的原始头。

如果代理不是HttpProxy或HttpCachingProxy的类型，则没有影响。

这个函数是在Qt 5.0中引入的。

See also QNetworkRequest::KnownHeaders, setHeader(), hasRawHeader(), and rawHeader().

void QNetworkProxy::setType( type)#

将这个实例的代理类型设置为type。

See also type() and setCapabilities().
