详细描述#

QSslConfiguration被Qt网络类用来转发关于一个开放的SSL连接的信息，并允许应用程序控制该连接的某些功能。

QSslConfiguration目前支持的设置有。

  • 要使用的SSL/TLS协议
  • 在连接过程中提交给对方的证书及其相关的私钥
  • 允许用于加密连接的密码器
  • 用于验证对等体证书的认证机构证书列表。

这些设置只在连接握手期间应用。在连接建立后设置它们没有任何影响。

QSslConfiguration支持的状态是。

  • 对等体在握手过程中出示的证书，以及通往CA证书的链条
  • 用于加密这个会话的密码

这些状态只能在SSL连接开始后获得，但不一定是在连接完成之前。在SSL连接的过程中，一些设置可能会发生变化，而不需要重新启动（例如，密码可能会随着时间的推移而改变）。

QSslConfiguration对象中的状态不能被改变。

QSslConfiguration可以与QSslSocket和网络访问API一起使用。

请注意，改变QSslConfiguration中的设置并不足以改变相关SSL连接中的设置。你必须在一个修改过的QSslConfiguration对象上调用setSslConfiguration来实现这一点。下面的例子说明了如何将QSslSocket对象中的协议改为TLSv1_0。

QSslConfiguration config = sslSocket.sslConfiguration();
config.setProtocol(QSsl::TlsV1_0);
sslSocket.setSslConfiguration(config);

See also QSsl::SslProtocol, QSslCertificate, QSslCipher, QSslKey, QSslSocket, QNetworkAccessManager, QSslSocket::sslConfiguration(), and QSslSocket::setSslConfiguration().

enum QSslConfiguration::NextProtocolNegotiationStatus#

描述了下一个协议协商（NPN）或应用层协议协商（ALPN）的状态。

Constant Value Description
QSslConfiguration::NextProtocolNegotiationNone 0 没有协商出任何应用协议（还没有）。
QSslConfiguration::NextProtocolNegotiationNegotiated 1 已经协商好了下一个协议（见nextNegotiatedProtocol()）。
QSslConfiguration::NextProtocolNegotiationUnsupported 2 客户端和服务器无法就共同的下一个应用协议达成一致。

[since 5.15]void QSslConfiguration::addCaCertificate(const &certificate)#

将certificate添加到这个配置的CA证书数据库。证书数据库必须在SSL握手之前设置。在握手阶段，CA证书数据库被套接字使用，以验证对方的证书。

注意：默认配置使用系统CA证书数据库。如果该数据库不可用（如iOS上常见的情况），默认数据库为空。

这个函数是在Qt 5.15中引入的。

参见 caCertificates(), setCaCertificates(), and addCaCertificates()。

[since 5.15]bool QSslConfiguration::addCaCertificates(const &path, [QSsl::EncodingFormat](qssl. html#EncodingFormat-enum) format = QSsl::Pem, [QSslCertificate::PatternSyntax] (qsslcertificate.html#PatternSyntax-enum) syntax = QSslCertificate::PatternSyntax::FixedString)#

在path中的所有文件中搜索以指定的format编码的证书，并将它们添加到此套接字的CA证书数据库。path必须是一个文件或匹配一个或多个文件的模式，如syntax所指定。如果一个或多个证书被添加到该套接字的CA证书数据库中，则返回 “true”；否则返回 “false”。

CA证书数据库在握手阶段被套接字使用，以验证对方的证书。

为了更精确地控制，请使用addCaCertificate()。

这个函数是在Qt 5.15中引入的。

参见 addCaCertificate() 和 QSslCertificate::fromPath() 。

<> QSslConfiguration::caCertificates() const#

返回此连接的CA证书数据库。CA证书数据库在握手阶段被套接字使用，以验证对方的证书。它可以在握手之前用setCaCertificates()修改，或者用addCaCertificate()和addCaCertificates()修改。

参见 setCaCertificates(), addCaCertificate(), 和 addCaCertificates()。

<>QSslConfiguration::ciphers() const#

返回这个连接当前的加密密码套件。这个列表在握手阶段用于选择一个会话密码。返回的密码列表是按降序排列的。(即，列表中的第一个密码是最受欢迎的密码）。会话密码将是列表中第一个也被对方支持的密码。

默认情况下，握手阶段可以选择该系统的SSL库所支持的任何密码，这可能因系统而异。本系统的SSL库支持的密码列表由supportedCiphers()返回。你可以通过调用setCiphers()来限制用于为该套接字选择会话密码的密码列表，其中包括一个支持的密码子集。你可以通过调用setCiphers()和supportedCiphers()返回的列表恢复到使用整个密码集。

注意：目前在Schannel后端不支持这个。

参见 setCiphers() 和 supportedCiphers()。

[static] QSslConfiguration::defaultConfiguration()#

返回在新的SSL连接中使用的默认SSL配置。

默认的SSL配置包括：。

  • 没有本地证书，没有私钥
  • 协议SecureProtocols
  • 系统默认的CA证书列表
  • 密码列表，相当于SSL库支持的128位或以上的SSL密码的列表

参见 supportedCiphers() 和 setDefaultConfiguration()。

[static] QSslConfiguration::defaultDtlsConfiguration()#

返回在新的DTLS连接中使用的默认DTLS配置。

默认的DTLS配置包括：。

  • 没有本地证书，没有私钥
  • 协议 DtlsV1_2OrLater
  • 系统默认的CA证书列表
  • 密码列表，等于SSL库支持的TLS 1.2密码的列表，该密码使用128或更多的秘密位。

又见 setDefaultDtlsConfiguration()。

[since 5.5]<>QSslConfiguration::ellipticCurves() const#

返回这个连接的当前椭圆曲线列表。这个列表在握手阶段用于选择一个椭圆曲线（当使用椭圆曲线密码时）。返回的曲线列表是按照降序排列的（也就是说，列表中的第一条曲线是最受欢迎的）。

默认情况下，握手阶段可以选择本系统的SSL库所支持的任何一条曲线，这可能因系统而异。本系统的SSL库所支持的曲线列表由QSslSocket::supportedEllipticCurves()返回。

你可以通过调用setEllipticCurves()来限制用于为这个套接字选择会话密码的曲线列表，并使用支持的密码的一个子集。你可以通过调用setEllipticCurves()和QSslSocket::supportedEllipticCurves()返回的列表来恢复到使用整个密码集。

这个函数是在Qt 5.5中引入的。

参见 setEllipticCurves。

QSslConfiguration::peerCertificate() const#

返回对等体的数字证书（即你所连接的主机的即时证书），如果对等体没有分配证书，则返回一个空证书。

对等体的证书在握手阶段会被自动检查，所以这个函数通常用于获取证书以显示或用于连接诊断。它包含关于对等体的信息，包括其主机名、证书颁发者和对等体的公钥。

因为对等体证书是在握手阶段设置的，所以从连接到QSslSocket::sslErrors()信号、QNetworkReply::sslErrors()信号或QSslSocket::encrypted()信号的槽中访问对等体证书是安全的。

如果返回的是空证书，这可能意味着SSL握手失败，也可能意味着你连接的主机没有证书，还可能意味着没有连接。

如果你想检查对等体的完整证书链，可以使用peerCertificateChain()来一次获得所有证书。

参见 peerCertificateChain(), QSslSocket::sslErrors (), [QSslSocket::ignoreSslErrors](qsslsocket. html#ignoreSslErrors)(), QNetworkReply::sslErrors(), 和 QNetworkReply::ignoreSslErrors()。

<>QSslConfiguration::peerCertificateChain() const#

返回对等体的数字证书链，从对等体的即时证书开始，以CA的证书结束。

在握手阶段会自动检查对等体的证书。这个函数通常用于获取证书以显示，或者用于执行连接诊断。证书包含关于对等体和证书颁发者的信息，包括主机名称、颁发者名称和颁发者的公共密钥。

因为对等体证书是在握手阶段设置的，所以从连接到QSslSocket::sslErrors()信号、QNetworkReply::sslErrors()信号或QSslSocket::encrypted()信号的槽中访问对等体证书是安全的。

如果返回一个空的列表，可能意味着SSL握手失败，也可能意味着你所连接的主机没有证书，也可能意味着没有连接。

如果你想只获得对等体的即时证书，请使用peerCertificate()。

参见 peerCertificate(), QSslSocket::sslErrors(), [QSslSocket::ignoreSslErrors](qsslsocket. html#ignoreSslErrors)(), QNetworkReply::sslErrors(), 和 QNetworkReply::ignoreSslErrors()。

int QSslConfiguration::peerVerifyDepth() const#

返回对等体证书链中在SSL握手阶段要检查的最大证书数量，如果没有设置最大深度，则返回0（默认值），表示要检查整个证书链。

证书按发行顺序检查，从对等体自己的证书开始，然后是其发行者的证书，以此类推。

参见 setPeerVerifyDepth()和peerVerifyMode() 。

QSslConfiguration::peerVerifyMode() const#

返回验证模式。该模式决定QSslSocket是否应该向对等体请求证书（即客户向服务器请求证书，或服务器向客户请求证书），以及是否应该要求该证书有效。

默认模式是AutoVerifyPeer，它告诉QSslSocket对客户端使用VerifyPeer，对服务器使用QueryPeer。

参见 setPeerVerifyMode（）。

QSslConfiguration::sessionCipher() const#

返回套接字的密码cipher，如果连接没有被加密，则返回空密码。套接字的会话密码是在握手阶段设置的。该密码用于加密和解密通过套接字传输的数据。

SSL基础设施还提供了设置密码有序列表的功能，握手阶段将最终从中选择会话密码。这个有序列表必须在握手阶段开始之前就已经到位。

[自5.2] QSslConfiguration::sessionTicket() const#

如果QSsl::SslOptionDisableSessionPersistence被关闭，这个函数会返回在SSL握手中使用的ASN.1格式的会话票，适合例如被持久化到磁盘。如果没有使用会话票或者QSsl::SslOptionDisableSessionPersistence没有被关闭，这个函数返回一个空的QByteArray。

注意：在将会话票据持久化到磁盘或类似的地方时，要注意不要将会话暴露给潜在的攻击者，因为对会话的了解可以窃听到用会话参数加密的数据。

这个函数是在Qt 5.2中引入的。

参见 setSessionTicket(), QSsl::SslOptionDisableSessionPersistence, setSslOption()和 QSslSocket::newSessionTicketReceived()。

[自5.2]int QSslConfiguration::sessionTicketLifeTimeHint() const#

[自6.0]void QSslConfiguration::setCiphers(const &ciphers)#

将此配置的密码套件设置为ciphers，这是一个用冒号分隔的密码套件名称的列表。这些密码是按照偏好的顺序排列的，从最偏好的密码开始。例如。

QSslConfiguration tlsConfig = QSslConfiguration::defaultConfiguration（）。
tlsConfig.setCiphers(QStringLiteral("DHE-RSA-AES256-SHA:DHE-DSS-AES256-SHA:AES256-SHA")。

ciphers中的每个密码名称必须是supportedCiphers()返回的列表中的一个密码的名称。限制密码套件必须在选择会话密码的握手阶段之前完成。

注意：目前在Schannel后端不支持这个功能。

这个函数是在Qt 6.0中引入的。

又见 ciphers()。

[自6.0]void QSslConfiguration::setHandshakeMustInterruptOnError(bool interrupt)#

如果interrupt为真，并且底层后端支持这个选项，那么在证书验证过程中发现的错误将立即通过发出QSslSocket::handshakeInterruptedOnError()来报告。这允许停止未完成的握手，并向对等体发送一个适当的警报信息。在这种情况下，应用程序不需要采取特别的行动。QSslSocket将在发送警报信息后关闭连接。如果应用程序在检查完错误后想要继续握手，它必须从其槽函数中调用QSslSocket::continueInterruptedHandshake()。信号-槽的连接必须是直接的。

**注意:**当中断握手被启用时，本来由QSslSocket::peerVerifyError()报告的错误反而只由QSslSocket::handshakeInterruptedOnError()报告。

注意：即使握手继续进行，这些错误也会在发出QSslSocket::sslErrors()信号时被报告（因此在相应的函数槽中必须被忽略）。

这个函数是在Qt 6.0中引入的。

参见 handshakeMustInterruptOnError(), QSslSocket::handshakeInterruptedOnError(), 和 QSslSocket::continueInterruptedHandshake ()。

void QSslConfiguration::setLocalCertificate(const &certificate)#

将SSL握手过程中提交给对等体的证书设置为certificate。

一旦连接建立，设置证书没有任何作用。

证书是SSL过程中使用的识别手段。本地证书被远程端用来对照其认证机构列表验证本地用户的身份。在大多数情况下，如在HTTP网页浏览中，只有服务器向客户识别，所以客户不发送证书。

又见 localCertificate()。

[since 5.1]void QSslConfiguration::setLocalCertificateChain(const <>&localChain)#

将SSL握手过程中提交给对等体的证书链设置为localChain。

一旦连接建立，设置证书链没有任何作用。

证书是SSL过程中使用的识别手段。本地证书被远程端用来对照其认证机构列表验证本地用户的身份。在大多数情况下，如在HTTP网页浏览中，只有服务器向客户识别，所以客户不发送证书。

与QSslConfiguration::setLocalCertificate()不同，这个方法允许你指定任何需要的中间证书，以便验证你的证书。列表中的第一项必须是叶子证书。

这个函数是在Qt 5.1中引入的。

又见 localCertificateChain()。

[自6.0]void QSslConfiguration::setMissingCertificateIsFatal(bool cannotRecover)#

如果cannotRecover为真，并且使用的验证模式是QSslSocket::VerifyPeer或QSslSocket::AutoVerifyPeer（对于客户端套接字），缺少同伴的证书将被视为一个不可恢复的错误，不能被忽略。在关闭连接之前，将向对等体发送一条适当的警告信息。

注意：只有当Qt被配置和构建为OpenSSL后端时才可用。

这个函数是在Qt 6.0中引入的。

See also QSslSocket::ignoreSslErrors(), QSslSocket::PeerVerifyMode, and missingCertificateIsFatal().
