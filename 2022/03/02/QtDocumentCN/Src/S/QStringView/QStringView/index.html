详细描述#

QStringView 引用它不拥有的 UTF-16 字符串的连续部分。它充当各种 UTF-16 字符串的接口类型，无需先构造QString。

UTF-16 字符串可以表示为QChar的数组（或数组兼容的数据结构，例如QString、std::basic_string 等），或者（在 Windows 等平台上，它是 16位类型）。ushort``char16_t``wchar_t

QStringView 被设计成一个接口类型；它的主要用例是作为函数参数类型。当 QStringViews 用作自动变量或数据成员时，必须注意确保引用的字符串数据（例如，由QString拥有）在所有代码路径上都超过 QStringView，以免字符串视图最终引用已删除的数据。

当用作接口类型时，QStringView 允许单个函数接受多种 UTF-16 字符串数据源。一个接受 QStringView 的函数因此替换了三个函数重载（采用QString和(const QChar*, int)），同时允许将更多的字符串数据源传递给函数，例如u"Hello World"字符串char16_t文字。

QStringViews 应该通过值传递，而不是通过引用到常量：

void myfun1(QStringView sv);        // preferred
void myfun2(const QStringView &sv); // compiles and works, but slower

如果你想让你的用户在他们可以传递给你的函数的字符串方面有最大的自由，请伴随 QStringView 重载和重载

  • QChar：这个重载可以委托给 QStringView 版本：

    void fun(QChar ch) { fun(QStringView(&ch, 1)); }

    即使出于技术原因，QStringView 本身也无法提供 QChar 构造函数。

  • QString：如果您存储了未修改的字符串副本，因此想利用 QString 的隐式共享。

  • QLatin1String：如果不转换QLatin1String 就可以实现功能 首先转为 UTF-16； 用户期望在 QLatin1String 上重载的函数执行的内存分配比 QStringView 版本的语义等效调用严格更少，涉及构建 QString 来自 QLatin1String。

QStringView 也可以用作函数的返回值。 如果您调用返回 QStringView 的函数，请特别注意不要使 QStringView 保持的时间长于函数承诺保持引用的字符串数据处于活动状态的时间。 如果有疑问，通过调用 toString() 将 QStringView 转换为 [QString](https ://doc.qt.io/qt-6/qstring.html）。

QStringView 是Literal Type，但由于它将数据存储为char16_t，因此迭代不是constexpr（从const char16_t* 转换为const QChar*，这在constexpr 函数中是不允许的）。 您可以在 constexpr 上下文中使用索引循环和/或 utf16()。

See also QString.

成员函数文档#

[since 6.0]template <typename Needle, typename Flags> decltype((*this, std::forward(needle), flags…)) QStringView::tokenize(Needle &&sep, Flags… flags) const#

在出现 sep 的地方将字符串拆分为子字符串视图，并返回这些字符串的惰性序列。

相当于

return QStringTokenizer{std::forward<Needle>(sep), flags...};

除非它在编译器中未启用 C++17 类模板参数推导 (CTAD) 的情况下工作。

请参阅 QStringTokenizer 了解 sep 和 flags 如何交互以形成结果。

注意： 虽然此函数返回 QStringTokenizer，但您永远不应该明确命名它的模板参数。 如果您可以使用 C++17 类模板参数推导 (CTAD)，您可以编写

QStringTokenizer result = sv.tokenize(sep);

（没有模板参数）。 如果不能使用 C++17 CTAD，则必须仅将返回值存储在 auto 变量中：

auto result = sv.tokenize(sep);

这是因为 QStringTokenizer 的模板参数对返回它们的特定 tokenize() 重载具有非常微妙的依赖性，并且它们通常不对应于使用的类型 为分隔符。

此功能是在 Qt 6.0 中引入的。

See also QStringTokenizer and qTokenize().

[since 6.2]qsizetype QStringView::lastIndexOf( l1, cs = Qt::CaseSensitive) const#

[since 6.2]qsizetype QStringView::lastIndexOf( str, cs = Qt::CaseSensitive) const#

此函数重载 lastIndexOf()。

返回此字符串视图中最后一次出现的字符串视图 str 或 Latin-1 字符串 l1 的索引位置，从该字符串视图的最后一个字符开始向后搜索。 如果没有找到 str，则返回 -1。

如果 cs 是 Qt::CaseSensitive （默认），则搜索区分大小写； 否则搜索不区分大小写。

这个函数是在 Qt 6.2 中引入的。

See also QString::lastIndexOf().

[since 5.14]qsizetype QStringView::lastIndexOf( c, qsizetype from = -1, cs = Qt::CaseSensitive) const#

[since 5.14]qsizetype QStringView::lastIndexOf( l1, qsizetype from, cs = Qt::CaseSensitive) const#

[since 5.14]qsizetype QStringView::lastIndexOf( str, qsizetype from, cs = Qt::CaseSensitive) const#

返回此字符串视图中最后一次出现的字符串视图 str、Latin-1 字符串 l1 或字符 ch 的索引位置，从索引位置 from 向后搜索。 如果 from 为 -1，则从最后一个字符开始搜索； 如果 from 为 -2，则在倒数第二个字符处，依此类推。 如果没有找到 str，则返回 -1。

如果 cs 是 Qt::CaseSensitive （默认），则搜索区分大小写； 否则搜索不区分大小写。

注意： 当搜索长度为 0 的 str 或 l1 时，数据末尾的匹配会被否定的 from 排除在搜索之外，即使通常认为是 -1 的 as 从字符串的末尾搜索：末尾的匹配是在最后一个字符之后，所以它被排除在外。 要包含这样一个最终的空匹配，要么给 from 一个正值，要么完全省略 from 参数。

这个函数是在 Qt 5.14 中引入的。

See also QString::lastIndexOf().

<> QStringView::toUcs4() const#

将字符串的 UCS-4/UTF-32 表示形式返回为 QList。

UCS-4 是一种 Unicode 编解码器，因此它是无损的。此字符串中的所有字符都将在 UCS-4 中编码。此字符串中任何无效的代码单元序列都将替换为 Unicode 替换字符（QChar::ReplacementCharacter，对应于 U+FFFD）。

返回的列表不是以 0 结尾的。

另请参阅 toUtf8()、toLatin1()、toLocal8Bit() 和 QStringEncoder。

QStringView::toUtf8() const#

以 QByteArray 形式返回字符串的 UTF-8 表示形式。

UTF-8 是一种 Unicode 编解码器，可以表示 Unicode 字符串中的所有字符，例如 QString。

另请参阅 toLatin1()、toLocal8Bit() 和 QStringEncoder。

[since 5.14]qsizetype QStringView::toWCharArray(wchar_t *array) const#

将此字符串转录到给定的 array 中。

调用者负责确保 array 足够大以容纳此字符串的 wchar_t 编码（分配与字符串长度相同的数组总是足够的）。该数组在 wchar_t 为 2 字节宽的平台上以 UTF-16 编码（例如 Windows）；否则（Unix 系统），wchar_t 被假定为 4 字节宽并且数据以 UCS-4 写入。

注意： 此函数不会将空终止符写入 array 的末尾。

返回写入 array 的 wchar_t 条目的数量。

这个函数是在 Qt 5.14 中引入的。

另见 QString::toWCharArray()。
